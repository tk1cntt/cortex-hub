# ADR-001: Architecture Pattern — Modular Monorepo

**Status:** Accepted
**Date:** 2026-03-18

## Context

Cortex Hub needs to host 3 applications (MCP gateway, Dashboard API, Dashboard Web) and Docker services (Qdrant, GitNexus). The team is 1 developer. The system targets < 1K users initially.

## Decision

**Modular Monorepo** using pnpm workspaces + Turborepo.

```
cortex-hub/
├── apps/
│   ├── hub-mcp/         → Docker container (Hono Node.js)
│   ├── dashboard-api/   → Hono + SQLite (self-hosted)
│   └── dashboard-web/   → Next.js 15 (Cloudflare Pages)
├── packages/
│   ├── shared-types/    → TypeScript interfaces
│   ├── shared-utils/    → Common utilities
│   └── ui-components/   → Shared React components
└── infra/               → Docker Compose + scripts
```

## Consequences

- ✅ Single repo for all code — simplified CI/CD, easier refactoring
- ✅ Shared types ensure type safety across apps
- ✅ Turborepo caches builds — fast CI
- ✅ Easy to extract services later if scale demands
- All services including hub-mcp run as Docker containers on the same host
- ⚠️ Dashboard API and Docker services require server access

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|----------------|
| Microservices repo-per-service | Over-engineered for solo dev, operational overhead |
| Single monolith app | Dashboard and MCP gateway have different deployment targets |
| Polyrepo | Cross-package type sharing is critical; polyrepo makes this painful |

# ADR-002: Communication Pattern — REST + WebSocket

**Status:** Accepted
**Date:** 2026-03-18

## Context

Three communication patterns needed:
1. Agent → Hub MCP: structured MCP protocol
2. Dashboard Web → Dashboard API: standard CRUD
3. Dashboard Web → Logs: real-time streaming

## Decision

- **MCP protocol** for agent ↔ hub communication (JSON-RPC over HTTP)
- **REST** (Hono) for Dashboard API (`/api/v1/*`)
- **WebSocket** for real-time log streaming to Dashboard
- **No message broker** in v1 — direct HTTP calls between services

## Consequences

- ✅ MCP protocol is standard — any MCP client works
- ✅ REST is simple, well-understood, cacheable
- ✅ WebSocket enables real-time logs without polling
- ⚠️ No async processing — all calls are synchronous in v1
- ⚠️ If log volume is high, may need SSE fallback

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|----------------|
| gRPC inter-service | Overkill for 3 services, adds build complexity |
| Redis Pub/Sub for logs | Extra dependency, WebSocket is simpler |
| GraphQL for Dashboard | REST is simpler for CRUD; GraphQL adds complexity |

# ADR-003: Data Strategy — SQLite + Vector + Graph

**Status:** Accepted
**Date:** 2026-03-18

## Context

Multiple data workloads:
- Application data (API keys, quality reports, logs) → relational
- Semantic search (knowledge, code) → vector
- Relationship traversal (memory links, code graph) → graph

## Decision

**Polyglot persistence:**

| Workload | Database | Rationale |
|----------|----------|-----------|
| App data | SQLite (WAL mode) | Zero-config, embedded, sufficient for single-server |
| Vectors | Qdrant | Purpose-built vector DB, fast HNSW, filtering |
| Code index | GitNexus (Docker eval-server) | AST parsing and code intelligence via HTTP API |

## Consequences

- ✅ Each DB optimized for its workload
- ✅ SQLite is zero-ops, no separate process needed
- ✅ Qdrant runs in Docker with persistent volumes
- ⚠️ 2 different query interfaces (SQL, Qdrant API)
- ⚠️ No cross-DB transactions (acceptable for this use case)

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|----------------|
| PostgreSQL for everything | Can't replace vector and graph workloads efficiently |
| Supabase | Cloud dependency, adds vendor lock-in for self-hosted goal |
| DuckDB | Better for analytics, not for transactional app data |

# ADR-004: Auth Architecture — API Key + OAuth2

**Status:** Accepted
**Date:** 2026-03-18

## Context

Two auth contexts:
1. **Agents → MCP**: machine-to-machine, needs to be simple and fast
2. **Humans → Dashboard**: interactive login, needs to be convenient

## Decision

- **API Key authentication** for MCP agent access
  - Keys stored hashed in SQLite
  - Per-key permissions (tool allowlists)
  - Per-key rate limits (default: 1000 req/hr)
  - Usage tracking per key
- **GitHub OAuth2** for Dashboard human login
  - OAuth2 code flow with PKCE
  - Session stored as httpOnly cookie
  - Single admin user in v1 (no RBAC)

## Consequences

- ✅ API keys are the industry standard for M2M auth — no agent SDK changes needed
- ✅ GitHub OAuth is familiar, provides avatar/username automatically
- ✅ Per-key permissions enable granular access control
- ⚠️ Single admin — no multi-user support in v1
- ⚠️ API key rotation requires agent config update

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|----------------|
| JWT tokens for agents | Adds token refresh complexity; API keys are simpler for M2M |
| OpenAI OAuth for agents | Not standard; OpenAI OAuth is for user accounts, not agent auth |
| Passkey/WebAuthn | Over-engineered for single admin dashboard |

# ADR-005: Multi-Tenancy — Single-Tenant (v1)

**Status:** Accepted
**Date:** 2026-03-18

## Context

v1 targets a single developer/team. Multi-tenancy is a v2+ concern.

## Decision

**Single-tenant** architecture. All data shares one SQLite database and one Qdrant collection. Agent isolation is via agent_id column, not separate databases.

## Consequences

- ✅ Simpler implementation — no tenant routing, no data partitioning
- ✅ Lower resource usage — single instance per service
- ⚠️ Cannot serve multiple independent teams in v1
- ⚠️ Retrofitting multi-tenancy requires schema migration (row-level isolation is planned)

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|----------------|
| Schema-per-tenant | Over-engineered for v1 single-tenant use case |
| DB-per-tenant | Massive overhead for SQLite + Docker services |
